FSP Overview

The software architecture of this sample program for RA Family MCUs is shown in the figure. FSP manages MCU-specific register settings and interrupt management for peripherals such as ADC, GPT, and SCI. It is a software package that includes the functions of the Hardware Abstraction Layer (HAL) as well as middleware functions such as a file system and communication protocol stacks.

Advanced Motor Platform overview

Advance Motor Middleware compatible with FSP 6.4 supports two types of methods: sensorless vector control and Hall sensor vector control. The control method can be switched by the Sensor module. The other modules are used in common.

Control Module

Overview

The Control module (rm_motor_control) is the top-level module that manages the lifecycle of the entire motor control system. It has an internal state machine (three states: STOP / RUN / ERROR) and centrally manages state transitions such as motor start/stop, error handling, and reset.

The Control module calls the lower-layer Algorithm module (PM FOC) to control motor driving. It also works with the Trajectory module to generate and manage speed command values. It also receives callbacks notified from the Algorithm layer (speed control completion, current control completion, error detection, etc.) and re-notifies the application layer.

How to Use

  1. Initialize the module with the open API. Internally, open of the Algorithm module and the Trajectory module are called.
  2. Start the motor with the run API (state transition STOP → RUN).
  3. Set the speed command value via the speedSet API or via the Trajectory module.
  4. Stop the motor with the stop API (state transition RUN → STOP).
  5. You can check the error state with the errorCheck API. When an error is detected, it transitions to the ERROR state and PWM output is stopped.
  6. You can return from the error state to the STOP state with the reset API.
  7. Terminate the module with the close API.

Algorithm (PM FOC) Module

Overview

The Algorithm (PM FOC) module (rm_motor_pm_foc) implements field-oriented control (FOC) for permanent magnet synchronous motors (PMSM). It consists of two control loops: an inner loop (current control) and an outer loop (speed control).

Inner loop (current control)
Driven by the ADC conversion complete interrupt (for 3-shunt/2-shunt methods) or the GPT underflow interrupt (for the 1-shunt method), and performs the following processing: - Acquisition of three-phase currents and dq coordinate transformation (Clarke-Park transform) - Calling the Sensor module to estimate angle and speed - PI control of d-axis and q-axis currents - Non-interference control (decoupling) - Voltage vector limitation - Sample delay compensation (optional) - Inverse transformation from dq coordinates to uvw coordinates - Voltage error compensation

Outer loop (speed control)
Driven by a timer interrupt and performs the following processing: - Speed feedback processing (selectable from speed observer, LPF, and extended observer) - Speed PI control - MTPA (maximum torque/current) control (optional) - Flux weakening control (optional)

Between the inner loop and the outer loop, data is exchanged using a double-buffer shared data structure to ensure data consistency between the two loops operating at different control cycles.

How to Use

  1. Initialize the module with the open API. Internally, initialization of the inner loop and outer loop, initialization of the Sensor module, and initialization of the HAL Driver module are performed.
  2. Activate the inner loop and outer loop with the run API and enable the HAL Driver.
  3. You can dynamically update control parameters (PI control gains, etc.) with the parameterUpdate API.
  4. Stop control with the stop API and disable PWM output.
  5. You can obtain error information for the inner loop and outer loop with the errorGet API.
  6. Terminate the module with the close API.

Sensor Module

Overview

The Sensor module is a module for estimating/detecting the motor’s rotation angle and speed. It conforms to the Motor Sensor API interface, and you switch between the following two implementations depending on the control method.

Induced-voltage observer sensorless vector method:

  • Sensorless vector control is a function for controlling a PM motor without a sensor to detect the magnetic pole position of the PM motor.
    An induced voltage (BEMF) observer is used for estimation. During stop and low speed, open-loop control using current pull-in is used. Once the speed reaches a level where induced voltage can be estimated, it switches to sensorless vector control. The switching speed differs depending on motor characteristics.
    Use an rm_motor_sensor_bemf instance in the sensor slot.

  • Parameter settings:
    For detailed configuration methods, check the parameter description of the rm_motor_sensor_bemf instance.
    Motor parameters are automatically passed from the motor parameter settings of rm_motor_pm_foc. You need to set the bandwidth of the BEMF observer, the bandwidth of the PLL, and the open-loop startup parameters.

Hall vector method: - Hall vector control is a function that detects speed and position using three Hall sensors and performs vector control.
Because Hall sensors are low-cost and have coarse resolution, speed is calculated from the interval of Hall sensor signal changes, and continuous angle estimation is performed by interpolating the angle between signal changes. This enables vector control (FOC) without using an encoder.

  • Parameter settings:
    For detailed configuration methods, check the parameter description of the rm_motor_sensor_hall instance.
    Motor parameters are automatically passed from the motor parameter settings of rm_motor_pm_foc. You need to set the Hall sensor input ports, Hall patterns, angle correction values, etc.

How to Use

  1. In the FSP stack settings, add the Sensor module to be used (BEMF or Hall) to the sensor slot of the PM FOC module.
  2. Initialize the module with the open API (automatically called during initialization of the PM FOC module).
  3. With the angleSpeedInit API, set the output pointers for angle/speed and the shared data structure with the inner loop.
  4. The angleSpeedCalc API is called for each control cycle of the inner loop to estimate angle and speed.
  5. With the positionSpeedGet API, you can obtain the mechanical angle position and speed (used from the outer loop).
  6. You can dynamically update control parameters with the parameterUpdate API.
  7. For the BEMF method, adjustment of open-loop startup parameters (Id current value, switching speed, etc.) is required.
  8. For the Hall sensor method, setting of Hall sensor input ports, Hall patterns, and angle correction values is required.

Driver Module

Overview

The Driver module (rm_motor_hal_driver) is a HAL (Hardware Abstraction Layer) module that controls the hardware required for motor driving (PWM timer, ADC).

Its main functions are as follows: - PWM output control: Calculates PWM duty ratios from three-phase voltage command values and sets them to the GPT timer. As modulation methods, it supports sine-wave PWM (SPWM), space-vector PWM (SVPWM), and two-phase modulation (DPWM). - Current detection: Detects three-phase currents and inverter DC bus voltage using the ADC. As current detection methods, it supports 3-shunt, 2-shunt, and 1-shunt methods. - Current offset correction: Automatically measures and corrects the offset of current detection at startup. - Bootstrap charging: Charges the bootstrap circuit at startup. - Enable/disable PWM output: Enables/disables PWM output by controlling the output of GPT pins. In the event of an error, PWM output is stopped immediately. - Saturation detection: Manages a saturation flag when the PWM duty ratio reaches upper/lower limits.

In the 1-shunt method, three-phase currents are reconstructed from a single shunt resistor by using the GPT double-buffer function and adjusting ADC trigger timing.

Inverter Settings

You need to set the inverter to be used and the operating conditions.
The default is set to use MCI-LV-1, as follows:

  • Carrier cycle: 20 kHz (10 kHz for the 1-shunt method)
  • Dead time: 0.5 us
  • DC bus voltage: 24 V

Inverter settings differ depending on the inverter used. If you change the inverter you use, refer to the manual and configure appropriate settings.

3shunt

The 3shunt current detection method provides shunt-based current detection on the low arm for each of the U, V, and W phases, and performs current detection using the valley interrupt of the PWM carrier.

1shunt

The 1shunt current detection method provides one shunt resistor on the low arm of the inverter, and reconstructs three-phase currents by performing multiple AD conversions by adjusting the timing of the PWM carrier.

The 1-shunt method has advantages such as reducing component cost on the board, but it has weaknesses such as processing time constraints and a tendency for noise, etc. to occur in the current waveform, resulting in a trade-off in control performance.

How to Use

In Advance Motor Middleware, the Trajectory module is used to pass speed commands to the motor control unit. If you do not use the Trajectory module, you can also set the speed command value directly with the speedSet API. In the current version, only speed control is supported.

  1. In the FSP stack settings, configure the Three-Phase PWM (GPT) and ADC instances, and connect them to the Driver module.
  2. Initialize the module with the open API (automatically called during initialization of the PM FOC module).
  3. Enable PWM output with the activate API. After current offset measurement is complete, the PWM output pins are automatically enabled.
  4. Set three-phase voltage command values with the phaseVoltageSet API (called from the inner loop).
  5. Obtain the detected three-phase currents, DC bus voltage, and maximum voltage vector with the currentGet API.
  6. Disable PWM output with the deactivate API.
  7. Set the following parameters according to the inverter specifications:
    • Carrier frequency (default: 20 kHz; 10 kHz for the 1-shunt method)
    • Dead time (default: 0.5 μs)
    • Modulation method (default: SVPWM)
    • Shunt resistor value, current detection amplifier gain, voltage detection gain
    • Voltage error compensation table

Trajectory Module

Overview

The Trajectory module generates and manages speed command values. In the current version, linear trajectory generation (Trajectory Linear) is supported.

The Trajectory module generates a smooth speed command profile by specifying the target speed and slew rate (acceleration). It also supports command queuing, allowing you to reserve the next command while the current trajectory is being executed.

How to Use

  1. In the FSP stack settings, add the Trajectory module to the Control module.
  2. In the Trajectory module properties, set “Use custom trajectory” to true, and in the application-layer C file, change the speed command value to a non-zero value.
  3. Build and run.
  4. If you do not use the Trajectory module, you can also set the speed command value directly with the speedSet API.

Tuner Module

Overview

The Tuner module provides an automatic estimation (parameter identification) function for motor parameters. Parameters such as motor resistance, inductance, and back-EMF constant can be automatically measured and estimated on actual hardware.

When the Tuner module is enabled, processing of the inner loop and outer loop is delegated to the Tuner, and control of PWM output is performed by the Tuner. When parameter identification is complete, you can obtain the estimated motor parameters.

Error Handling

Overview

In this library, error detection is performed in the inner loop (current control) and the outer loop (speed control). When an error is detected, PWM output is stopped immediately and the application layer is notified through a callback function.

Error List

Error Description
Overcurrent (HW) Hardware (POEG) overcurrent detection
Overcurrent (SW) Software overcurrent detection of phase current
Overvoltage Overvoltage detection of inverter DC bus voltage
Undervoltage Undervoltage detection of inverter DC bus voltage
Overspeed Overspeed detection of mechanical angular speed
Flux weakening abnormality Abnormality detection of flux weakening control

Recovery from Errors

When an error occurs, you can reset from the error state and return to the stop state by calling the reset API. After that, you can start the motor again with the run API.

State Machine

The Control module is managed by a state machine with the following three states.

State Description
STOP Stop state. The motor is not rotating.
RUN Run state. The motor is rotating.
ERROR Error state. An error has been detected and the motor is stopped.

State transitions occur due to the following events.

Event STOP → RUN → ERROR →
RUN RUN RUN ERROR
STOP STOP STOP ERROR
ERROR ERROR ERROR ERROR
RESET STOP ERROR STOP

Callback Functions

The Control module calls callback functions for the following events.

Event Description
SPEED_BACKWARD When processing of speed control (outer loop) is completed
CURRENT_BACKWARD When processing of current control (inner loop) is completed
ERROR When an error is detected

Register callback functions in the settings of the Control module (rm_motor_control) instance.
